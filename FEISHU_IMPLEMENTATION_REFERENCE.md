# 飞书集成实现参考

这份文档只保留跨项目可复用的关键逻辑、实现分层和落地流程，适合在新项目里作为飞书接入参考。具体业务字段、项目名称、当前表结构细节等都不在这里展开。

## 1. 总体设计

建议把飞书能力拆成两层，而不是把所有逻辑揉进一个模块：

- 通知层：负责把任务结果、异常、日报等内容发到飞书群
- 表格层：负责把业务数据写入飞书 Sheets，作为台账、状态中心或展示视图

这样拆分的好处：

- 通知和表格互不依赖，可以单独启用
- 业务逻辑不会和飞书 API 细节强耦合
- 后续迁移到别的通知渠道或别的表格存储时，改动范围更小

## 2. 飞书通知实现模式

### 配置建议

- Webhook 地址优先从环境变量读取
- 配置文件作为兜底
- 运行时允许显式关闭通知，避免本地调试误发消息

### 发送流程

推荐固定成下面这条链路：

1. 收集本次任务摘要
2. 组装成 Markdown 或卡片内容
3. 调用飞书机器人 Webhook
4. 校验响应状态
5. 失败时记录日志并允许重试

### 实现建议

- “内容构建”和“发送请求”分成两个函数
- 不要在业务主流程里直接拼接通知文本
- 通知函数尽量只接受结构化数据，而不是零散字符串

## 3. 飞书表格实现分层

建议把表格部分拆成三层。

### API Client 层

职责：

- 获取访问令牌
- 发送 HTTP 请求
- 读写表格、sheet、range
- 处理分页、限流、鉴权失败

这一层尽量只关心“怎么和飞书 API 说话”，不要掺杂业务字段。

### Workbook 层

职责：

- 确保工作簿和 sheet 存在
- 维护表头
- 把二维数组和字典列表互转
- 提供整表覆盖、分页读取、单行 upsert 等通用能力

这一层是最值得复用的部分。很多项目的飞书接入，真正稳定与否，取决于有没有这层包装。

### Projection / Sync 层

职责：

- 把业务领域对象投影成一个或多个 sheet
- 控制哪些字段写入展示表，哪些字段写入事实表
- 决定同步顺序和回写规则

这一层才应该理解“业务状态是什么”。

## 4. 认证模式选择

飞书接入通常有两种常见模式。

### `app` 模式

适合：

- 后台服务
- 定时任务
- 机器人或自动化系统

特点：

- 用应用凭证换 token
- 不依赖人工登录
- 更适合长期稳定运行

### `user` 模式

适合：

- 明确和个人账户绑定的数据访问
- 需要代表某个用户操作的场景

特点：

- 常常需要用户授权
- 权限更贴近个人账号
- 对自动化任务来说维护成本更高

### 选择建议

- 如果项目是自动化系统，优先选 `app`
- 如果必须代表个人身份访问用户私有资源，再考虑 `user`

## 5. 工作簿初始化策略

不要每次启动都盲目新建工作簿或 sheet，推荐做幂等初始化。

### 推荐做法

- 用一个本地状态文件保存：
  - spreadsheet token
  - spreadsheet url
  - 各个 sheet 的 id
- 启动时按下面顺序初始化：
  1. 先读本地已保存的 id
  2. 如果找不到，再按 sheet 标题在远端查找
  3. 还找不到才创建新的 sheet
  4. 创建完成后立刻回写状态文件

### 这样做的价值

- 避免重复创建同名 sheet
- 降低初始化时的 API 调用量
- 让同步逻辑可重复执行

## 6. 推荐的表格模型

不要把“历史数据、当前状态、人工展示”混在一张表里。

更稳的做法是至少拆成三类：

### 事实表

一行表示一个最小业务事实，例如一次尝试、一次状态、一次关联关系。

适合：

- 程序回写
- 调度读取
- 做后续统计

### 展示表

一行表示一个更高层的业务对象，方便人工查看。

适合：

- 汇总视图
- 人工筛选
- 运营查看

### 历史表

存历史迁移结果、旧系统导入事实、补录记录等。

适合：

- 追溯来源
- 与旧系统对账
- 迁移期做兼容

## 7. 写入策略

飞书表格通常有两类写入方式，各自适合不同场景。

### 整表覆盖

适合：

- 展示表
- 汇总视图
- 每次都由程序全量重建的表

优点：

- 逻辑简单
- 不容易残留旧数据
- 很适合投影表

缺点：

- 数据量大时成本更高
- 不适合高频细粒度回写

### 单行 Upsert

适合：

- 运行时状态更新
- 单条任务回写
- 高频更新的事实表

优点：

- 更新成本低
- 适合流式写入

缺点：

- 需要设计稳定主键
- 如果匹配规则不稳，容易写出重复行

### 实践建议

- 展示层优先整表覆盖
- 事实层优先单行 upsert
- upsert 一定要基于“规范化主键”匹配，而不是原始文本直接比对

## 8. 历史迁移经验

历史数据接入时，不要直接把旧字段原样搬到新表。

推荐流程：

1. 先把旧数据转成标准化事实
2. 再由事实推导当前状态
3. 最后把当前状态投影到展示表

### 迁移时最常见的问题

- URL 有多种格式，需要统一规范化
- 飞书富文本链接和纯字符串链接格式不一致
- 旧系统里用的短标识、新系统里想换成更清晰的标识

### 处理建议

- 先做 URL 规范化层
- 兼容飞书富文本 URL 解析
- 旧标识到新标识做一层映射表

## 9. 标识设计经验

如果一个对象会长期存在、还要跨系统流转，建议优先使用更稳定、更可读的标识。

例如：

- 域名
- slug
- 外部系统唯一 id

不要过度依赖只在当前项目里看得懂的短别名。  
短别名在早期开发阶段方便，但一旦表变大、站点增多、涉及迁移或审计，就会越来越难维护。

## 10. 最小复用方案

不同项目可以按复杂度选择不同接入深度。

### 方案 A：只接通知

适合：

- 只想把任务成功/失败发到飞书群
- 还不需要飞书表格做状态中心

### 方案 B：通知 + 轻量台账

适合：

- 想在飞书里保留任务记录
- 需要人工查看运行结果

### 方案 C：完整状态中心

适合：

- 飞书要作为业务事实源或核心后台
- 需要调度、回写、迁移、展示分层

## 11. 落地清单

新项目接飞书时，建议先把下面这些问题确认清楚：

1. 飞书只做通知，还是也做业务事实源
2. 认证模式选 `app` 还是 `user`
3. 是否需要本地状态文件保存工作簿和 sheet id
4. 哪些表是事实表，哪些表是展示表
5. 哪些写入用整表覆盖，哪些写入用 upsert
6. 是否存在旧系统数据，需要单独做历史标准化
7. 主键和标识是否足够稳定、可读
8. 所有 URL 是否都有规范化和富文本兼容处理

## 12. 关键实践结论

最值得在别的项目里直接复用的经验有四条：

- 通知层和表格层要拆开
- 飞书 API 之上最好再包一层 Workbook 抽象
- 历史数据、当前状态、展示视图要分层
- 所有主键、URL、外部标识都要先规范化，再做 upsert 或迁移

如果一个新项目能先把这几件事搭好，后面的飞书接入通常会稳定很多，也更容易扩展。
